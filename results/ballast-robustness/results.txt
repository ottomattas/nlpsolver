BALLAST ROBUSTNESS: DISTRACTOR SENTENCES AT INCREASING DOSE -- plain-text summary
=================================================================================
Relevant dir: https://github.com/ottomattas/nlpsolver/tree/main/results/ballast-robustness
Full results with all detail and caveats: https://github.com/ottomattas/nlpsolver/blob/main/results/ballast-robustness/results.md


Setup: nlpsolver/llmpipe on macOS (local gk ARM64), current two-stage pipeline, LLM cache on, thinking off, prover default 2s.
THE ASK (Tanel, June 10): insert pure-ballast sentences (no shared content words, no synonym/antonym links even soft) into the test cases at increasing dose; find when accuracy starts to degrade; spot-check the generated clause sets.
STATE SO FAR: all four doses (b2/b4/b8/b16) run on the 100-subset with gpt+claude. The dose effect is clear and the failure mode is informative. CAVEAT: post-run auditing caught a generator text bug that contaminated a minority of cases (worst at b16) plus an Anthropic credit outage inside claude-b16, so the Phase-2 numbers below are PROVISIONAL; the generator is fixed, clean suites are committed, and a clean rerun (~$55-60) awaits go-ahead. The 1600 runs are NOT started.

WHAT I RAN
==========

G  Ballast generator. Pool = the statement sentences of the 1600 set itself (738 unique after dropping questions, wh-sentences, pronoun sentences, duplicates). A sentence is accepted as ballast for a case only if it shares NO content words with it (raw + morphological variants incl. irregulars + canonical forms), has no ANTONYMS link, no SOFT_SYNONYMS link at any score, no shared exclusion group, and no pair in two different noun-mutex groups; ballast sentences are also mutually inert. Insertion slots (start / between / right before the question) from a per-(case,dose) seeded RNG; deterministic output; self-checks re-verify everything from the written files.
   script: https://github.com/ottomattas/nlpsolver/blob/main/llmpipe/tests/ballast/make_ballast.py
   suite:  https://github.com/ottomattas/nlpsolver/blob/main/llmpipe/tests/ballast/tests_core_100_b2.py
   manifest (ballast + sources + slots per case): https://github.com/ottomattas/nlpsolver/blob/main/llmpipe/tests/ballast/tests_core_100_b2.manifest.json
   -> All 100 cases generated at the STRICT level (no relaxation needed; min 196 eligible pool sentences per case).

S  Clause-set spot checks ("pistelisi kontrolle"). Original vs ballasted runs, full pipeline, then a clause-level diff: no clause may mix ballast and original vocabulary, no injected axiom may bridge the two sides, the answer must stay correct. Probes included a wh-case, a defeasible case, a measure case and a pronoun case.
   script: https://github.com/ottomattas/nlpsolver/blob/main/llmpipe/tests/ballast/spot_check.py
   -> The check CAUGHT a real gap: inject_isa_cross_group_axioms bridged ballast "house" with original "animal" (cross-group noun mutex) on the wh-case. Added that injector's own lexicon as a generator condition; after the fix all six probes pass clean. Exactly the kind of thing you warned about.

1  Phase 1: 100-subset at b2, gpt + claude (200 runs, 0 errors).
   data: https://github.com/ottomattas/nlpsolver/tree/main/results/ballast-robustness/core_100_b2
   analysis: https://github.com/ottomattas/nlpsolver/tree/main/results/ballast-robustness/analysis
   -> gpt 100 -> 98, claude 98 -> 96. Two ballast sentences already hurt slightly, but only slightly.

2  Failure decomposition. Every single b2 failure is "no proof" (logic too weak -> Unknown); zero wrong answers, zero errors. Re-running all 5 lost cases with -seconds 10 (5x prover budget) changes nothing -> these are NOT prover timeouts.
   script: https://github.com/ottomattas/nlpsolver/blob/main/results/ballast-robustness/analysis/dose_response.py
   -> IT SEEMS LIKE the degradation enters ONLY through the LLM parse: the encoding silently loses clauses the proof needs. The symbolic half is indifferent to inert ballast, as it should be. Lost cases (553 comparative, 893 possessive presupposition, 22 conditional, 1500/1521 hedged confidence) overlap with the fragile classes from the parsing-architecture study.

3  Ballast handling audit over all 200 runs: did the models parse the ballast, and did any parse MERGE ballast with original content?
   script: https://github.com/ottomattas/nlpsolver/blob/main/results/ballast-robustness/analysis/ballast_audit.py
   -> Ballast was parsed in all 200 runs (never skipped). Found a small COREFERENCE-CAPTURE failure mode: claude resolved ballast "This was deep." onto the original "John is glad." and lost case 22; gpt re-bound "She" to a ballast NP on case 134 (survived) and merged "Dr." + "Penguins" into one entity on 1408 (survived). For the next doses the pool filter will also drop demonstrative-subject sentences (pronouns are already dropped); at b2 this cost exactly one case.

4  Position effect (start vs middle vs right-before-question): at this dose too few failures to say anything (gpt's two losses had start-placed ballast, claude's three involved before-question placement -- opposite directions, anecdote level). Phase 2 doses will give this power.
   script: https://github.com/ottomattas/nlpsolver/blob/main/results/ballast-robustness/analysis/position_effect.py

5  Token/cost ledger. llmcall.py now captures each real API call's usage field into the per-case JSONs (additive change), so spend is measured, not estimated.
   change: https://github.com/ottomattas/nlpsolver/blob/main/llmpipe/solver/llmcall.py
   script: https://github.com/ottomattas/nlpsolver/blob/main/results/ballast-robustness/analysis/token_ledger.py
   -> Phase 1 cost $7.71 at list prices (gpt $2.35, claude $5.36): provider prompt caching served ~97.8% of input tokens, so the run came in well under the ~$15 estimate. Phase 0 (key smoke + spot checks) ~$0.7. Session total ~$8.4.

6  Phase 2: the full dose ladder b4/b8/b16 on the 100-subset, gpt+claude (600 runs), at a non-binding 32K output budget (the default 8K cap mechanically truncates b16 parses -- an operational finding by itself).
   data: https://github.com/ottomattas/nlpsolver/tree/main/results/ballast-robustness (core_100_b4 / b8 / b16)
   exclusions: https://github.com/ottomattas/nlpsolver/blob/main/results/ballast-robustness/phase2_exclusions.json
   -> The dose effect is now unambiguous (table below). Failures stay parse-side (no_proof dominates at every dose), but from b4 on a WRONG-ANSWER bucket appears (the parse produces misleading clauses, not just fewer), and claude twice emitted syntactically malformed logic at b8/b16. gpt's b4 (85) < b8 (95) is genuinely non-monotonic -- each dose has an independent ballast draw -- the clean rerun will show if it is draw-specific interaction or decode variance.

7  Post-run audit caught TWO data-validity problems; both are handled and documented:
   a) Sentence-splitter bug: "A surgeon, Dr. Smith, entered the room." was split at "Dr.", which put fragments into the ballast pool and even let ballast be inserted INSIDE that sentence in one case. The earlier "Dr. Penguins" curiosity from the b2 audit was THIS bug (corrupted suite text), not LLM entity-merging -- correction to item 3. Contaminated: 2/6/7/32 cases at b2/b4/b8/b16; excluded from the valid-cases numbers below. Fix + cleanly regenerated suites: https://github.com/ottomattas/nlpsolver/blob/main/llmpipe/tests/ballast/make_ballast.py
   b) Anthropic credits ran out mid-b16: 22 consecutive claude cases got no API response at all and count as failures in the raw numbers; excluded in the valid-cases numbers. Credits need topping up before any claude rerun.

READING: two ballast sentences barely hurt; by b16 both models lose roughly a fifth of the cases they used to solve. Failures are overwhelmingly "logic too weak -> Unknown" (the prover is never the problem; 5x prover budget changes nothing), plus a small wrong-answer bucket from b4 on. N_light looks like b4-b8, N_heavy like b16 -- to be confirmed on the clean rerun before committing to the 1600 runs.

Cost so far (usage ledger, list prices): Phase 0 ~$0.7; Phase 1 (b2) $7.71; Phase 2 (b4+b8+b16) $46.78. Total ~$55. Caching keeps marginal cost ~4x under list-price estimates (~97.8% of input tokens served from provider prompt caches).
